# docs-audit 품질 루프

`docs-audit`는 이제 단순한 오래된 문서 정리 스킬이 아니라, 문서 구조와 품질을 함께 보는 문서 건강검진 스킬로 사용한다.

## 보는 것

- 문서가 지금도 유효한가
- 다른 문서와 중복이 큰가
- 허브 / 심화 / 사례 / 레퍼런스 역할이 분명한가
- 관련 문서 링크가 자연스럽게 이어지는가
- 실무 학습 가치와 면접 가치가 있는가
- 최근 diff가 실제 개선인지, 단지 문장을 옮긴 것인지

## 추천 사용 시점

### 1. 문서가 꽤 쌓였을 때
예:
- `mysql` 문서군 구조가 복잡해졌을 때
- `spring` 문서군이 횡단 관심사, 트랜잭션, JPA 축으로 늘어났을 때

### 2. Claude maintainer / writer가 큰 정리를 한 직후
이때는 반드시 diff-validation 모드로 본다.

질문 예시:
- `최근 mysql 정리 diff 기준으로 실제로 좋아졌는지 봐줘`
- `spring 문서 품질 평가해줘`

### 3. 주기 점검
최근 생성 문서 2~5개 또는 특정 폴더 하나를 골라 품질 점검한다.

## 운영 원칙

- 바로 삭제하지 않는다
- 먼저 `keep / refresh-needed / merge / archive / delete-candidate`로 분류한다
- 최근 문서는 공격적으로 정리하지 않는다
- 허브와 심화 문서는 역할이 분명하면 중복으로 간주하지 않는다

## 현재 확인된 유용한 패턴

- MySQL 인덱스 문서군은 허브 + 심화 분리가 실제로 품질 개선 효과가 있었다
- 오래된 학습 노트는 내용이 유효해도 현재 품질 기준에서는 `refresh-needed`로 판정될 수 있다
- README / 허브 문서가 있으면 새 문서가 늘어도 역할 충돌을 줄이기 쉽다
