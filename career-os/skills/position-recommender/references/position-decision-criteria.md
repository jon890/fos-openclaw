# 포지션 추천 의사결정 축

이 문서는 포지션 추천을 반복할수록 판단 기준을 업그레이드하기 위한 기준점이다. 매일 추천 결과나 사용자의 피드백에서 새로운 판단 축이 나오면 여기에 추가한다.

## 핵심 목표

- NHN보다 커리어/보상/브랜드/엔지니어링 규모 면에서 나은 회사로 이동한다.
- 기존 게임/사내 플랫폼 도메인에 갇히지 않고, 결제·핀테크·커머스·검색·포털·모빌리티·콘텐츠·물류·클라우드·플랫폼·AI product 등으로 적극 전환한다.
- 기본 지원 축은 Java/Spring 서버/백엔드다. AI/AX/MLOps는 후보자의 실제 AI workflow, RAG, Agent orchestration 경험이 서버 개발 역할과 결합될 때만 우선순위를 높인다.

## 우선순위 판단 축

회사/규모 업사이드의 세부 기준은 `config/company-upside-reference.md`를 함께 참조한다. 포지션 추천 시 JD fit만 보지 말고, NHN 대비 브랜드/보상/트래픽/도메인/엔지니어링 규모 업사이드를 반드시 평가한다.

### 1. JD 핏

- Java/Spring, Kafka, Redis, MySQL/RDBMS, OpenSearch, Batch, MSA, 운영 트러블슈팅과 직접 연결되는 포지션을 우선한다.
- Kotlin/Node/Python/Kubernetes/MLOps 중심 포지션은 강점 전이가 분명할 때만 stretch로 둔다.
- Data Engineer, ML Engineer, AI Research, Frontend, PM, QA, Designer는 사용자가 명시하지 않는 한 제외한다.

### 2. 차별화 가능성

- 단순히 “할 수 있는 일”보다, 후보자의 고유 경험이 지원자 풀 안에서 눈에 띄는지를 본다.
- 예: 결제 서비스 백엔드는 Outbox/정합성/트랜잭션 경험과 잘 맞지만, 비교군도 결제 백엔드 경험자일 수 있다.
- 예: AX/AI Transformation 서버 포지션은 AI workflow, Agent orchestration, RAG batch, 자동화 파이프라인 경험으로 차별화 가능성이 더 클 수 있다.

### 3. 회사/도메인 업사이드

- NHN 대비 브랜드, 보상, 트래픽 규모, 엔지니어링 문화, 성장성, 도메인 확장성을 함께 본다.
- 네이버/카카오/LINE 계열, 쿠팡, 카카오페이/카카오뱅크/카카오모빌리티, 우아한형제들, 당근, 오늘의집, 무신사, 야놀자, 컬리 등은 JD 핏이 맞을 때 우선 검토한다.
- 대기업/유명 회사라도 JD 핏이 약하면 억지 추천하지 않는다.

### 4. 역할 구성

- “서버 개발자” 이름이 붙어 있어도 실제 역할이 제품 서버/플랫폼 서버 개발인지, 내부 도구 운영/기획/조율 중심인지 확인한다.
- AX 포지션은 서버 개발 비중이 충분하고 전사 주요 프로젝트/AI 전환 파이프라인 설계 역할이면 우선순위를 높인다.
- AX가 단순 업무툴 도입, 프롬프트 운영, 내부 지원 업무 중심이면 결제서비스 같은 정통 백엔드 포지션을 더 안전하게 본다.

### 5. 지원 이력/쿨다운

- 최근 6개월 내 지원 후 불합격 경험이 있는 회사/계열사는 하드필터로 삭제하지 않고, 우선순위를 낮추는 쿨다운 신호로 본다.
- 현재 쿨다운 대상: 토스 계열 전체(토스/비바리퍼블리카/토스뱅크/토스페이먼츠/토스플레이스).
- 쿨다운 대상도 JD 핏이 매우 좋으면 “보류/쿨다운 후보”로 짧게 언급할 수 있다. 다만 강력 추천/이번 주 즉시 지원 액션에는 올리지 않는다.
- 쿨다운 대상의 기술 블로그나 사례는 학습 참고자료로 사용할 수 있다.

### 6. 하드필터와 소프트 랭킹 구분

- 하드필터는 최소화한다. 기본적으로는 공고를 삭제하기보다 감점 사유와 우선순위로 표현한다.
- 하드필터 대상: 마감/폐쇄 공고, 계약직/임시직/프리랜서/위촉직/인턴, 서버/백엔드와 무관한 직무, 명백히 필수조건 미충족인 포지션.
- 소프트 랭킹 대상: 최근 탈락 회사, Kotlin/Python/K8s 등 일부 기술 갭, 도메인 전환 난이도, 회사 업사이드 불확실성, 내부도구/조율 비중 우려.
- 소프트 랭킹 대상은 “보류/주의” 또는 “도전 추천”으로 남겨 전체 시장 감각을 유지한다.

### 7. 고용 형태와 신선도

- 정규직/상시 정규직만 강력 추천/도전 추천에 올린다.
- 계약직, 임시직, 프리랜서, 위촉직, 인턴은 추천·보류 후보에서 제외한다.
- 마감/폐쇄 공고를 추천하지 않는다. Wanted는 가능하면 API status=active를 확인한다. 카카오 계열은 public API/list 또는 신뢰 가능한 현재 노출 근거를 확인한다.

## 현재 적용 중인 세부 판단

### 카카오페이 AX vs 결제 서비스

- 카카오페이 AX가 “서버 개발자 - 전사 주요 프로젝트 및 AX 주도”처럼 실제 서버 개발 역할이고, 전사 주요 프로젝트/AI Transformation/업무 방식 재설계에 깊게 들어가는 포지션이면 결제서비스보다 우선한다.
- 이유: 후보자의 AI workflow, Agent orchestration, RAG batch, Confluence ADF 변환/검색 파이프라인, Cursor Rules/Claude Code 기반 개발 프로세스 개선 경험이 AX에서 더 선명하게 차별화된다.
- 결제 서비스는 Java/Spring, 대용량 결제 데이터, 트랜잭션, 동시성, 분산 트랜잭션, 대사 배치, MSA, 결제 API, VAN/카드사 연동과 후보자의 Outbox/Batch/정합성 경험이 매우 잘 맞는 안전한 강력 후보다.
- 단, AX의 제품 서버/플랫폼 개발 비중이 낮거나 내부 도구/운영 지원 중심이면 결제서비스를 더 안전한 1순위로 본다.

## 추천 결과 작성 원칙

- 추천은 단순 순위가 아니라 “왜 이 포지션이 지금 더 나은 의사결정인지”를 설명한다.
- 각 포지션마다 다음을 함께 제시한다.
  - 직접 링크
  - JD 핏
  - 후보자 경험 근거
  - 차별화 포인트
  - NHN 대비 업사이드
  - 위험/갭
  - 바로 할 다음 액션
- 사용자가 피드백한 판단 축은 다음 추천부터 반영하고, 필요하면 이 파일을 업데이트한다.
