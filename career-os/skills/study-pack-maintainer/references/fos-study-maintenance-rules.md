# fos-study 유지보수/연결 규칙 레퍼런스

이 문서는 fos-study의 `CLAUDE.md`와 `.claude/skills/*`를 바탕으로, 기존 문서를 검토하고 새 문서/업데이트/보강/브리지 전략을 고를 때 따라야 할 규칙을 정리한다.

## 1. 우선 판단 순서
1. 기존 문서 업데이트로 충분한가?
2. 기존 문서에 실제 사례 매핑만 보강하면 되는가?
3. 두 문서 이상이 이미 좋아서 허브/브리지 문서가 더 자연스러운가?
4. 위 셋이 아니면 새 문서 생성

## 2. 문서 역할 분리
- `task/` 문서는 실제 업무 경험/구현 사례 기록
- `architecture/`, `database/`, `java/` 등은 개념 문서
- 개념 문서가 사례를 흡수할 수는 있지만, 사례 자체를 통째로 대체하지는 않는다
- 사례 문서는 개념 문서를 링크하고, 개념 문서는 사례 문서를 인용/연결하는 구조를 선호한다

## 3. 사례 매핑 보강 기준
아래 중 하나 이상이면 `augment-existing`를 우선 검토한다.
- 일반 개념 문서는 좋은데 실제 경험 연결이 약함
- 문제 해결 흐름은 있는데 JVM/GC/운영 대응처럼 해석 레이어가 약함
- 면접 답변에서 꺼낼 수 있는 실제 사례가 부족함
- 기존 문서를 갈아엎을 필요는 없고, 실전성만 올리면 됨

## 4. 브리지 문서 생성 기준
아래 조건이면 `create-bridge`를 검토한다.
- 두 문서 이상이 각자 역할이 분명하고 품질도 좋다
- 억지로 하나로 합치면 범위가 흐려진다
- 독자가 '이걸 같이 봐야 이해된다'는 연결이 분명하다
- 상위 허브나 비교/연결 문서가 있으면 학습 경로가 좋아진다

## 5. 민감 정보와 일반화
- 업무 사례 문서에서는 내부 식별자/고유 비즈니스 명칭을 일반화할 수 있는지 항상 검토
- 공개 가능한 기술 구조와 공개하면 안 되는 사업 자산을 구분
- 문서 보강 시 새 예시를 넣더라도 같은 기준을 따른다

## 6. 문체/품질
- AI 요약체보다 실제로 해본 사람이 정리한 문체를 선호
- '왜 그렇게 판단했는지'와 trade-off를 분명히 적는다
- 본문은 학습 가능성과 면접 활용성을 동시에 만족해야 한다
- 관련 문서가 있으면 상대 경로 링크를 적극 사용한다
